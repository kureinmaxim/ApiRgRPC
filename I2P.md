## 1. Что это такое (в одном абзаце)

**I2P (Invisible Internet Project)** — это анонимная оверлейная сеть («сеть поверх
сети»): полностью одноранговый, зашифрованный слой поверх обычного интернета, в
котором адресуются не по IP, а по **криптографическим адресам**. Трафик идёт
через цепочки промежуточных узлов (туннели), на каждом шаге заново шифруется, так
что ни один узел не видит одновременно и отправителя, и получателя, и содержимое.
I2P оптимизирован для **скрытых сервисов внутри сети** (eepsites, p2p-приложения),
а не для выхода в обычный интернет — это его ключевое отличие от Tor.

## 2. Происхождение и история

- **Корни (≈2000–2002).** Идея выросла из сообщества **Freenet** и проекта
  **IIP (Invisible IRC Project)** — анонимного IRC. Нужен был обобщённый
  анонимный транспортный слой, не привязанный к одному приложению.
- **Запуск (2003).** Проект стартовал как **I2P** под руководством разработчика
  под псевдонимом **jrandom**. Реализация — на **Java** (этот клиент так и
  называют «Java I2P», официальный референс).
- **Смена поддержки (≈2007).** jrandom отошёл от проекта; разработку продолжило
  сообщество (заметные участники под никами **zzz**, **str4d** и др.). Проект
  выжил без единого «владельца» — это децентрализованное opensource-сообщество.
- **i2pd (≈2013 →).** Появилась независимая реализация на **C++** —
  **i2pd** (автор под ником **orignal**). Лёгкая, без JVM, идеально для серверов,
  роутеров, встраивания. **Именно i2pd мы используем** в проекте.
- **Развитие протокола.** Транспорты эволюционировали: старые **NTCP/SSU** →
  современные **NTCP2** и **SSU2** (улучшенные шифрование и устойчивость к DPI).
  Криптография сместилась к **Ed25519/X25519** (тип адресов «EdDSA», ключи нового
  формата).
- **Применение в реальных проектах.** Историческая интеграция с **Monero**
  (проект **Kovri**, затем i2pd) для приватности транзакций; различные
  p2p-мессенджеры, файлообмен (i2psnark — torrent внутри I2P), форумы и почта
  внутри сети.

## 3. Как это работает (модель)

### 3.1 Garlic routing («чесночная маршрутизация»)
Развитие идеи onion routing (Tor): сообщения («cloves» — «зубчики») **связываются
в пучок** («garlic» — «чеснок») и шифруются слоями. Каждый промежуточный узел
снимает свой слой и видит только следующий хоп. Связывание нескольких сообщений в
один пучок усложняет анализ трафика.

### 3.2 Туннели (tunnels)
- I2P строит **однонаправленные** туннели: отдельно **исходящие** (outbound) и
  **входящие** (inbound). Для диалога A↔B задействованы как минимум 4 туннеля.
- Туннель — цепочка из нескольких роутеров (по умолчанию длина 2–3 хопа). Туннели
  **короткоживущие** (≈10 минут), постоянно перестраиваются → меняющийся путь.
- Это отличается от Tor, где цепь (circuit) двунаправленная.

### 3.3 NetDB и floodfill
Адреса и маршрутная информация (**RouterInfo**, **LeaseSet**) хранятся в
распределённой базе **NetDB** (Kademlia-подобная DHT). Часть узлов —
**floodfill** — держат и раздают эту базу. **Нет центральных directory
authority**, как в Tor → меньше точек контроля, но дольше «прогрев» сети.

### 3.4 Адресация
- **`.i2p`-домены** (человекочитаемые, через локальную адресную книгу) и
  **b32-адреса** — `<52 символа base32>.b32.i2p`, производные от криптоключа
  сервиса. b32 самодостаточен (не требует адресной книги) — **именно его мы
  передаём как `peers`/destination** в конфиге RNS.
- Скрытый сервис = **Destination** (пара ключей). Поднял сервис → получил b32.

### 3.5 SAM API (то, что использует Reticulum)
**SAM (Simple Anonymous Messaging)** — управляющий API i2pd/Java-I2P (порт по
умолчанию **7656**). Через него внешнее приложение просит I2P: «создай мне
destination», «установи соединение к такому-то b32», «прими входящие». **Reticulum
`I2PInterface` ходит именно через SAM** — поэтому на обоих концах нужен i2pd с
включённым SAM. Приложение (мост, клиент) само I2P-протокол не реализует — только
говорит с локальным i2pd по SAM.

## 4. I2P vs Tor (коротко)

| Признак | I2P | Tor |
|---|---|---|
| Назначение | скрытые сервисы **внутри** сети (eepsites, p2p) | анонимный доступ в **обычный** интернет (+ onion-сервисы) |
| Маршрутизация | garlic, **однонаправленные** туннели | onion, двунаправленные circuits |
| Каталог сети | распределённый **NetDB** (floodfill), без центра | **directory authorities** (полу-централизованно) |
| Выход в клирнет | через outproxy (не основной сценарий) | основной сценарий (exit nodes) |
| Размер сети | меньше | больше |
| Прогрев | медленный старт туннелей (десятки секунд) | быстрее |
| Наш сценарий | **да** — оба конца внутри I2P, публичный порт не нужен | — |

Для нас важно: I2P-сервис **не слушает TCP наружу** — мост перестаёт нуждаться в
открытом публичном порту (50061) и в SSH-туннеле. Это и есть мотивация этапа 3.

## 5. Перспективы

- **Устойчивость к цензуре и DPI.** NTCP2/SSU2 спроектированы как трудноразличимые;
  есть транспортные обфускации. Активно дорабатывается.
- **Децентрализация без directory authority** — нет очевидной «головы», которую
  можно отключить; привлекательно для resilient-инфраструктуры.
- **i2pd как встраиваемый слой.** Лёгкий C++-демон удобно ставить на VPS, роутеры,
  в контейнеры; растёт как «приватный транспорт по умолчанию» для p2p-проектов.
- **Связка с Reticulum.** Reticulum абстрагирует среду: один и тот же код моста и
  клиента работает поверх TCP, I2P, радио. I2P даёт «бесплатную» анонимность и
  обход NAT/файрволов без своей инфраструктуры — мощный задел на будущее.
- **Риски/ограничения.** Меньшая сеть → выше латентность и дольше установка
  первого соединения; меньше аудита, чем у Tor; outproxy-выход слаб (но нам он не
  нужен).

## 6. Практическая справка (i2pd)

### 6.1 Установка
**Linux (VPS, Debian/Ubuntu):**
```bash
apt update && apt install -y i2pd
systemctl enable --now i2pd
systemctl status i2pd --no-pager
```
**Windows (клиент):** установить i2pd (msi-инсталлятор с GitHub
`PurpleI2P/i2pd` releases) либо через пакетный менеджер; запустить службу/демон.

### 6.2 Включить SAM (нужен для Reticulum `I2PInterface`)
Конфиг i2pd: Linux — `/etc/i2pd/i2pd.conf`; Windows — `%APPDATA%\i2pd\i2pd.conf`.
```ini
[sam]
enabled = true
address = 127.0.0.1
port = 7656
```
После правки — перезапустить: `systemctl restart i2pd` (Linux).

### 6.3 Порты по умолчанию (локальные, на 127.0.0.1)
| Порт | Сервис | Зачем |
|---|---|---|
| **7656** | **SAM API** | **используется Reticulum I2PInterface** |
| 7070 | Web-консоль | статус, туннели, диагностика в браузере |
| 4444 | HTTP-прокси | сёрфинг eepsites (не нужно для нас) |
| 4447 | SOCKS-прокси | универсальный прокси (не нужно для нас) |
| 7654 | I2CP | «родной» клиентский протокол I2P |

> Наружу i2pd слушает **свой** случайный порт для NTCP2/SSU2 (его можно
> зафиксировать в `[ntcp2]/[ssu2] port`), но **управляющие порты (7656/7070)
> держим только на localhost.**

### 6.4 Полезные команды
```bash
systemctl status i2pd --no-pager           # жив ли демон
journalctl -u i2pd -n 100 --no-pager       # логи (тут же стартовые сообщения)
ss -tlnp | grep -E ':7656|:7070'           # SAM и web-консоль слушают?
curl -s http://127.0.0.1:7070/ | head      # web-консоль отвечает?
```
Web-консоль (`http://127.0.0.1:7070`) показывает: число известных роутеров,
построенные туннели, статус сети — удобно понять, «прогрелась» ли сеть.

### 6.5 Траблшутинг
| Симптом | Причина | Что делать |
|---|---|---|
| Долго нет соединения (десятки сек) | I2P строит туннели (30–120 c при холодном старте) | подождать; смотреть web-консоль (7070) — растёт ли число туннелей |
| Reticulum не видит I2P | SAM выключен/не на 7656 | проверь `[sam] enabled = true`, `ss -tlnp \| grep 7656` |
| 0 known routers в консоли | сеть не «прогрелась» / нет исходящего интернета | дать время, проверить, что i2pd достаёт reseed-серверы |
| Адрес b32 не находится | LeaseSet ещё не опубликован / неверный b32 | подождать, перепроверить b32 из логов моста |

### 7.1 Грабли и рабочие приёмы (наш опыт, 2026-06-26)

- **i2plib не нужен отдельно** — Reticulum вендорит его (`RNS.vendor.i2plib`).
  Проверка: `python -c "from RNS.vendor import i2plib; print('ok')"`.
- **Логи RNS под systemd были невидимы** — Python блочно буферизует stdout, когда
  он идёт в pipe journald (не TTY), и строки RNS застревали в буфере. Фикс —
  drop-in `Environment=PYTHONUNBUFFERED=1` для `ha-reticulum-bridge.service`:
  ```bash
  mkdir -p /etc/systemd/system/ha-reticulum-bridge.service.d
  printf '[Service]\nEnvironment=PYTHONUNBUFFERED=1\n' \
    > /etc/systemd/system/ha-reticulum-bridge.service.d/unbuffered.conf
  systemctl daemon-reload && systemctl restart ha-reticulum-bridge
  ```
- **b32 логируется на уровне INFO (4)** — в конфиге RNS было `loglevel = 3`
  (NOTICE), поэтому `endpoint ready … b32` не печатался. Поднять `loglevel = 4`
  в `/opt/TelegramOnly/ha_stack/rns/config` на время поиска b32.
- **Свежий i2pd прогревается медленно** — пока `Tunnel creation success rate`
  низкий (~25%) и `LeaseSets = 0`, destination создан, но не «ready» (нет inbound-
  туннелей). Помогает: высокий класс полосы (`bandwidth = P`/`X`) и время.
- **Где увидеть b32 до того, как RNS залогирует «ready»** — web-консоль i2pd
  `http://127.0.0.1:7070/?page=local_destinations` (или `?page=sam_sessions`).
- **Несколько b32 в Local Destinations** — «хвосты» от прошлых запусков RNS;
  активный соответствует ESTAB-сессии `python → 127.0.0.1:7656` (`ss -tnp | grep 7656`).
- **Диагностика «RNS↔SAM живой»**: `ss -tnp | grep 7656` (есть ESTAB от python),
  `curl -s http://127.0.0.1:7070/?page=local_destinations | sed 's/<[^>]*>/ /g'`.
